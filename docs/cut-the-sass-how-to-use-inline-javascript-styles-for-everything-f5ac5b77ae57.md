今天，前端 web 开发是关于构建应用程序，而不仅仅是网页。虽然 CSS 和 SASS 都非常适合样式化页面，但是它们并不适合样式化应用程序。如果您正在使用基于组件的策略进行构建，这一点尤其正确。

基于组件的策略是将应用程序分解成称为组件的小的、可重用的构建块。使用组件构建您的应用程序可以为您的前端带来可重用性、清晰性和效率。

为了从基于组件的方法中获得全部好处，我们希望组件完全封装功能、标记和样式。

在这篇文章中，我们将探索如何使用内联 JavaScript 编写风格是支持基于组件的开发的一个好方法。最终，我们将得到前端组件，它们是单个文件，易于共享，并且完全独立。

用 JS 编写你的风格是一个很大的改变，所以我想花一些时间来解释为什么我们想要这样做。我相信为正确的工作选择正确的工具。SASS 无疑是当时的正确工具，提供了一些巨大的好处，例如:

*   **嵌套**
*   **变数**
*   **Mixins**

# **变量和混合**

我们已经讨论了 SASS 的嵌套，那么其他大的好处呢——变量和混合？

好吧，恰好存在另一个非常流行的解决方案——JavaScript！

如果我们能够利用 JavaScript 的“var”和“function”的自然特性来提供我们在样式解决方案中需要的东西，这不是很好吗？如果我们使用 JavaScript 来编写我们的风格，我们会获得我们喜欢的 JS 的灵活性、功能和熟悉性。另外！我们的整个组件可以用一种语言编写！甚至一个文件！

## **我们的目标**

这是我们想要做的:

*   在标记旁边直接组合我们的样式
*   “导入”和“导出”变量，如颜色或断点
*   基于状态(打开/关闭菜单等)编写“动态”样式。)

这里还有一些要求:

*   需要覆盖伪状态，如`:hover，:before，:after `等
*   必须直接支持媒体查询，如` @media (max-width: 600px)`等
*   允许函数/术语(见上面的动态)

## ***旁注:解释“JS 样式”和“JS 中的 CSS”的区别***

有两种不同的方法:

*   您可以在 JS 中包含 CSS
*   你可以使用 JS 编写你的 CSS

我更喜欢第二种方法，这也是我们将在这里讨论的。

对于第一种方法， [CSS 模块](https://github.com/css-modules/)是“在你的 JS 中编写 CSS”类别中一个流行的解决方案。它限定了你写在组件的“div<div>/div>中的常规 CSS 的范围，或者命名空间。名称间距是您可以用类或 id 手动完成的事情，并不是单独使用这种东西的足够理由。对于可变支持或混合，您需要使用 CSS“增强器”,如 PostCSS 或带有 CSS 模块的 SASS。

同样，我们希望能够利用 JavaScript 的语言特性，包括变量、术语或函数。如果我们的解决方案迫使我们编写某种“增强的”CSS 而不是 JS，它将无法满足我们的需求。

有很多 css-in-js 解决方案可以用于这两种方法。看看他们！现在，我们将实现我发现的最能解决我们的目标和需求的方法。

# **结论**

那么，我们用这种方法做了什么呢？嗯，我们能够将一个相当复杂的组件的所有内容封装在一个文件中，并且使用一种语言。

因为实现本质上都是 JavaScript，所以您可以选择想要如何构造组件。JavaScript 允许的所有模式对您都是可用的。

这意味着您的样式解决方案可以直接反映您的整体应用程序架构。这种方法允许您完全控制自己的结构。您可以移动这些部分，并以您认为最容易阅读和最合理的方式将它们组合在一起。

根据我的经验，最好的方法是创建许多小的、完全隔离的组件；其中大部分是“无状态”或“愚蠢”的组件，只在所谓的[不可变应用架构](https://vimeo.com/album/3953264/video/166790294)中自顶向下接收数据。

我更喜欢这些组件封装它们的标记、样式和功能。当有一个单一的位置和一个清晰的依赖树时，维护或更改组件就变得微不足道了。

因为这些组件只有一个关注点，所以文件很少会变得太大。如果它们变得太复杂，您可以很容易地将它们抽象或重构为单独的组件。

内联 JS 样式的另一个好处是，在进行更新时定位代码变得很简单，因为组件所需的一切都在同一个文件中。

以这种方式组成的组件也易于共享，因为您可以冒充为单个组件。js 文件。你的消费者可以满足他们认为合适的“进口”申报。“进口反应”没有理由不能变成“进口角度”

我希望我已经展示了内联 JS 风格的强大之处，以及它们如何能够取代 SASS，并对基于组件的开发提供了一些见解。这是一种真正适合我们团队的方法，希望它也适合你的团队！
